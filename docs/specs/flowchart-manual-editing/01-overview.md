## 目的

会議中のメモ書きレベルで、SVGフローチャートに**手動編集機能**を追加する。
プロフェッショナルな図形編集ツールではなく、**軽量で直感的なメモ機能**として実装する。

## 設計方針

### 基本コンセプト

```
Mermaid自動生成フローチャート（論理的・美しい）
            ↓
      会議中に気づき発生
            ↓
     手動でエッジ・メモ追加（柔軟・即座）
            ↓
   接続関係をJSON/CSV出力（構造化データ）
            ↓
    タスク管理やドキュメント生成に活用
```

### 実装のスコープ

#### ✅ 実装する機能

| 機能 | 理由 | 優先度 |
|------|------|--------|
| **2点クリックで直線エッジ追加** | シンプルで実装コストが低い | 🔴 必須 |
| **ノード自動吸着（半径50px）** | ノード間接続の使い勝手向上 | 🟡 高 |
| **エッジラベル追加** | 関係性を明示できる | 🟡 高 |
| **接続関係のJSON/CSV出力** | データ活用の幅が広がる | 🟢 中 |
| **視覚的フィードバック** | UX向上 | 🟢 中 |
| **手動エッジの削除** | 誤操作の修正 | 🟡 高 |

#### ❌ 実装しない機能

| 機能 | 実装しない理由 |
|------|--------------|
| **曲線エッジ（Bézier）** | 複雑すぎる（制御点計算、レンダリング） |
| **自動レイアウト** | Dagreアルゴリズム等、数千行のコードが必要 |
| **ノード移動時のエッジ追従** | 座標管理が複雑化、保守コスト高 |
| **Mermaid記法への逆変換** | SVG → Mermaidは技術的に困難 |
| **Undo/Redo** | 実装コスト高、Phase 4以降で検討 |
| **メモノード追加** | Phase 2以降で検討（エッジのみで十分か評価） |

### 技術制約

#### 環境制約
- **外部ライブラリ不使用**: ブラウザ標準APIのみで実装
- **対象ブラウザ**: Microsoft Edge / Google Chrome 86+
- **ファイル操作**: File System Access API（ローカルファイル直接操作）
- **スタンドアロン動作**: サーバー不要、HTMLファイル単体で完結

#### 技術スタック
```
言語: JavaScript (ES6+)
UI: HTML5 + CSS3
SVG: SVG DOM API（直接操作）
ファイルI/O: File System Access API
データ形式: JSON, CSV
```

### データフロー

```
1. ユーザー操作
   ↓
2. エッジ追加モード切り替え
   ↓
3. SVG上でクリック（開始点）
   ↓
4. ノード検索 → 吸着判定
   ↓
5. マウス移動でプレビュー表示
   ↓
6. SVG上でクリック（終了点）
   ↓
7. ノード検索 → 吸着判定
   ↓
8. エッジラベル入力ダイアログ
   ↓
9. SVG DOMに<line>要素追加
   ↓
10. JavaScript内部データ（manualEdges）に保存
   ↓
11. JSONファイルに自動保存
```

## リスクと対策

### 技術リスク

| リスク | 影響 | 対策 |
|--------|------|------|
| SVG座標系の複雑さ | 高 | getBBox(), getScreenCTM()の活用 |
| ズーム時の座標変換 | 中 | viewBox座標系に統一 |
| ブラウザ互換性 | 低 | Edge/Chrome 86+に限定 |

### スコープリスク

| リスク | 影響 | 対策 |
|--------|------|------|
| 機能追加要求の増加 | 高 | Phase分けで段階的実装 |
| 既存機能との干渉 | 中 | イベントハンドラの優先順位管理 |
