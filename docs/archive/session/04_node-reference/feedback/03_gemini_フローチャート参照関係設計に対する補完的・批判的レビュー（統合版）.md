
# フローチャート参照関係設計に対する補完的・批判的レビュー

- 作成日: 2026-03-22
- 対象: `01_current-implementation.md` 〜 `06_technical-concerns.md` および既存設計
- 目的: 既存の設計を前提に、批判的かつ客観的な立場から不足しがちな論点を補うこと

---

## 1. 総合評価

これまでの整理と既存レビューは**全体として極めて妥当**です。  
特に、

- ノード吸着のような不安定な自動化から撤退したこと
- 人間が明示的に参照関係を管理する方針へ切り替えたこと
- SVGメタデータとして永続化しようとしていること
- 段階的実装（Phase分割）を意識していること

は、単一HTML・Vanilla JavaScript・外部依存なしという制約下では、非常に現実的で筋の良い判断です。

そのため、ここでは無理に否定するのではなく、**「この設計は概ね正しいが、実装・運用フェーズや将来の拡張で後から問題化しやすい点はどこか」**という観点で、補完的かつ批判的なフィードバックを行います。

---

## 2. 新たに追加したい批判的・客観的視点

### 2.1. 最大の論点は「参照関係」そのものより、**参照の意味が混ざっていないか**です

現状の議論では `references` が万能な関係フィールドとして扱われています。しかし実際には、少なくとも以下は意味が違います。

- **補足対象**: 「このメモノードはこの正規ノードの補足である」
- **関連対象**: 「この要素は別の要素と関連がある」
- **依存対象**: 「この要素は別の要素を前提としている」
- **追跡対象**: 「このタスクはこのノードに紐づく」

これらを全部 `references` で表すと、将来AIに渡したときの意味解釈が曖昧になったり、ハイライトやフィルタの仕様が増えたときに破綻しやすくなります。

**【提言】**
MVPでは `references: string[]` で問題ありません。ただし、設計文書上は**「`references` は現時点では『汎用の関連先ID一覧』であり、将来的に relation type を導入する可能性を前提とする」**と明記すべきです。また、タスク連携情報は `references` に吸収せず、別構造のまま維持する方が良いでしょう。

### 2.2. 過小評価されやすい最大リスクは **IDの安定性** です

Mermaid由来のIDや自動生成IDは、実装都合の識別子であって、必ずしも業務上安定した識別子ではありません。Mermaidを再生成したり、表示順が変わったりすると、ID対応が崩れる可能性があります。

**【提言】**
理想は、参照に **ID以外のヒント情報** を持たせることです（例: `targetHint: { label: "アーキテクチャ設計" }`）。MVPとしては `references: string[]` のままとしつつ、読込時に参照先IDが見つからない場合、**ラベルベースの再解決候補をユーザーに提示できる余地を残す**のが現実的です。

### 2.3. 本当に決めるべきなのは「どこが正本（Source of Truth）か」です

タスク連携をSVGに保存するかどうかの論点は、「正本はどこか」を決める問題です。

- A側: タスクマネージャー（LocalStorage）
- B側: フローチャートエディター（SVGメタデータ）

この状態で方針が曖昧だと後で必ず同期問題になります。

**【提言】**
現状の制約なら、**「タスク本体の正本はA側であり、SVG内の `taskFlowchartLinks` はスナップショット（バックアップ）である」**と明文化すべきです。競合時はA側を優先し、SVG単独読込時のみスナップショットを復元に使うルールを固定してください。

### 2.4. UI/UXの本質的な課題は「入力方法」より **運用コスト** です

タグUIが見やすいかより、**人間が更新し続けられるか**の方が重要です。運用が進むと入力が面倒になり、情報が古くなって信用されなくなるのが典型的な失敗パターンです。

**【提言】**
入力コストを下げるため、既存レビューにある「ラベル検索」「ピッキングモード」「手動エッジからの候補提示」を重視してください。さらに、**「どこにも参照していないメモノード一覧」**を出せるようにすると、運用改善に直結します。

### 2.5. エッジまで参照対象に含めるかは慎重に

設計上は全要素に共通で `references` を持たせるのは美しいですが、実務上はエッジ参照は使いどころが曖昧になりやすく、UI上でノイズになります。

**【提言】**
MVPでは参照先を**ノードとメモノードに限定**し、エッジ参照はPhase 3以降に開放するか、デフォルト非表示にすることを推奨します。

### 2.6. セキュリティ上の論点は XSS だけではありません

XSSやLocalStorage制限だけでなく、以下にも注意が必要です。

- **CSSセレクタインジェクション**: `document.querySelector('[data-id="${id}"]')` の `id` に特殊文字が入ると壊れます。`CSS.escape(id)` を使うべきです。
- **Prototype Pollution**: 外部SVG内のJSONをそのままオブジェクトにマージすると危険です。JSONをそのまま信用せず、ホワイトリスト検証や `Object.create(null)` の活用を検討してください。
- **SVG自体の危険性**: 読み込むSVGに `<script>` や `on*` 属性が含まれている場合のサニタイズ処理が不可欠です。

## 3. 長期運用・アーキテクチャ拡張に向けた補完的視点

ここでは、現状の設計には含まれていないが、将来必ず直面するであろう技術的な課題への先回りした提言を行います。

### 3.1. テスト可能性（Testability）の確保

単一HTML・Vanilla JSで開発を進めると、UI（DOM操作）とデータ操作（状態管理）が密結合しやすくなります。参照関係が複雑になると、手動テストでは限界が来ます。

**【提言】**
参照を追加・削除・検証する関数は、**DOMに依存しない純粋関数（Pure Functions）として分離**してください。
```javascript
// ❌ DOMに依存した悪い例
function addReference(id, refId) {
  elementReferences[id].push(refId);
  updateUI();
}

// ⭕️ 状態操作を分離した良い例
function computeNewReferences(currentRefs, newRefId) {
  if (currentRefs.includes(newRefId)) return currentRefs;
  return [...currentRefs, newRefId];
}
```
これにより、UIをブラウザで動かさなくても、ロジック単体の正しさをテストできるようになります。

### 3.2. データマイグレーション戦略

`schemaVersion` を導入する計画は素晴らしいですが、バージョンが `1.0` から `2.0`、さらに `3.0` へ上がった際の**変換ロジックの置き場所**を設計しておく必要があります。

**【提言】**
SVGメタデータを読み込んだ直後に動作する、マイグレーション専用のパイプラインを用意してください。
```javascript
function migrateMetadata(parsedData) {
  let data = structuredClone(parsedData);
  if (!data.schemaVersion || data.schemaVersion === "1.0") {
    data = migrateV1toV2(data);
  }
  if (data.schemaVersion === "2.0") {
    data = migrateV2toV3(data);
  }
  return data;
}
```

### 3.3. 削除・改名・複製の「意味論」の定義

参照関係は追加時より、**後から要素が変化したとき**に破綻しやすいです。実装に入る前に以下の仕様を固定しておくべきです。

- **削除時**: 参照は自動削除するが、削除前に「X件の参照元があります」と警告を出す。
- **複製時**: 参照は原則コピーしない（参照先が変わる可能性があるため）。
- **改名時**: ID参照であれば影響なし。ただしラベルベースで補助解決している場合は再評価が必要。

## 4. 実務的なアクションプランと実装ステップ

上記のレビューを踏まえ、Phase分割をより安全かつ確実な順序に再構成した提案です。

### Step 1: 参照モデルの固定と正本の定義（設計フェーズ）
- `references: string[]` の採用
- タスク連携は別構造
- 「正本はA側、SVGはスナップショット」のルール明文化

### Step 2: データ構造と最小UIの実装（Phase 1 + MVP）
- メタデータ保存・復元ロジックの実装（`CSS.escape` 等の安全対策を含む）
- UIでの「タグ表示（読み取り専用）」と「無効参照の警告」

### Step 3: 入力・編集UIの拡充（Phase 2）
- タグ入力とラベル部分一致検索（オートコンプリート）
- フローチャート上をクリックして追加する「ピッキングモード」
- 手動エッジからの候補提案機能

### Step 4: 破綻しやすいエッジケースの検証
- 別のSVGを上書き読込した際のID不一致テスト
- ノード削除・参照先消失のテスト
- タスク同期競合テスト
- 循環参照時の挙動テスト（探索深さ制限の導入）

### Step 5: 利便性・拡張機能（Phase 3以降）
- 逆参照の動的表示
- SVG上での参照線の可視化（破線描画など）
- Relation Type（関連の種類）の導入検討

---

## 結語

全体として、**「技術的に困難な自動化（吸着）より、人間が確実に管理できる明示的なデータ構造」**を選んだ判断は高く評価できます。

無理に大きく作り替える必要はありません。実装に入る前に、以下の1文だけでも設計書へ追記しておくことを強くお勧めします。

> *「本機能における `references` は図要素間の汎用的な関連先ID一覧として扱う。タスク連携は別ドメイン情報として独立管理する。また、Mermaid由来IDの不安定性を考慮し、長期的には補助的な再解決情報を持てる拡張余地を残すこととし、SVG内のタスクデータはあくまでスナップショットとして扱う。」*

この前提さえチーム（または自分自身）で共有できていれば、今後の実装上の迷いやバグの発生を劇的に減らすことができるはずです。
