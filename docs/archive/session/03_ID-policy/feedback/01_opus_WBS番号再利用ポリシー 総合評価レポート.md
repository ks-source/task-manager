### 2.2 データ整合性

| 観点 | 選択肢1 | 選択肢2 |
|------|---------|---------|
| 参照整合性（predecessors） | ✅ **完全に保持** — WBS番号が不変のため、参照先は常に存在する | ⚠️ **更新ロジック依存** — 更新漏れ = 参照破壊（しかもサイレント） |
| JSON E/I の安定性 | ✅ 単純な読み書き。既存データとの互換性100% | ⚠️ インポート時に `_del` の有無とdeleted flagの整合性チェック必要。旧バージョンJSONとの互換性にマイグレーション必須 |
| WBS番号一意性 | ✅ 全履歴から最大値を取得するため、重複は原理的に発生しない | ⚠️ `_del`を付与するため元番号空間は空くが、「同じ番号を再利用できる」という仕様自体がデータ混乱の源 |

**補足 — 選択肢2の「隠れたデータ整合性リスク」**:

提供資料では触れられていないが、以下のエッジケースが存在する：

```
1. TEST.15.1 を作成
2. TEST.15.1 を削除 → TEST.15.1_del
3. TEST.15.1 を再利用して新規作成
4. 新しい TEST.15.1 を削除 → TEST.15.1_del ← 名前が衝突！
```

このケースでは `TEST.15.1_del` が2つ存在することになり、WBS番号の一意性が完全に破壊される。対処するには `_del_1`, `_del_2` のような連番サフィックスが必要になり、複雑さが指数関数的に増大する。

## 3. 具体的な質問への回答

### Q1. ID再利用の是非

**回答: ID再利用は避けるべきです。**

ID（識別子）の再利用は、ソフトウェア工学における「ABA問題」と本質的に同じです。同じIDが異なるエンティティを指す状態は、あらゆるレイヤーで混乱を生みます。業界標準ツールの86%がID不変性を採用していることが、この原則の堅牢さを裏付けています。

特にWBS番号は**ユーザー可視ID**であり、人間が会話やドキュメントで参照します。「TEST.15.1の件ですが…」と言った時、それが削除前のタスクなのか再利用後のタスクなのか、文脈だけでは判別できません。

### Q2. WBS番号の特殊性

**回答: WBS番号は通常のシーケンシャルIDよりも厳格な不変性が求められます。**

理由：
- **意味を持つID**: `TEST.15.1`は「TESTフェーズの15番目のメジャータスクの1番目のサブタスク」という意味を内包する。変更すると意味構造が崩れる
- **階層構造**: 親子関係がIDに埋め込まれているため、IDの変更は構造情報の変更を意味する
- **口頭での参照**: WBS番号はミーティングやチャットで頻繁に言及される。「TEST.15.1 はキャンセルになったので TEST.16.1 に移行してください」は自然だが、「TEST.15.1 が TEST.15.1_del になりました」は混乱を招く

### Q3. 参照整合性の懸念

**回答: 選択肢2の参照破壊の懸念は妥当であり、かつ過小評価されています。**

提供資料では`predecessors`と`flowchart.nodeIds`のみが参照フィールドとして挙げられていますが、以下も考慮が必要です：

1. **ユーザーが手動で書いたコメント** — `comments`フィールド内で「TEST.15.1を参照」のようなテキストがある場合、自動更新は困難
2. **外部ドキュメント** — ユーザーがExcelやメール等でWBS番号を参照している場合、システム外の参照は更新できない
3. **将来追加されるフィールド** — `related_tasks`、`blocked_by`等が追加された場合、`updateTaskReferences()`の更新を忘れるリスクがある

選択肢1ではこれらすべてが**問題にならない**（IDが変わらないため）。

### Q4. トレーサビリティ

**回答: 選択肢1の方がトレーサビリティに優れています。**

一見すると選択肢2の`_del`サフィックスがトレーサビリティを向上させるように見えますが、実際にはそうではありません：

- **選択肢1のトレーサビリティ**: `deleted: true` + `deleted_at` で完全な監査証跡が残る。WBS番号が不変なので、過去の参照（ドキュメント、コミットメッセージ等）から元のタスクを確実に追跡できる
- **選択肢2のトレーサビリティ**: `_del`サフィックスはJSON内で目立つが、IDの変更自体が追跡を困難にする。「TEST.15.1」で検索しても`TEST.15.1_del`はヒットしない（完全一致検索の場合）

真のトレーサビリティとは「IDの不変性 + 状態の記録」であり、IDを変更することではありません。

### Q5. 代替案

**回答: 選択肢1を基盤とした段階的拡張を提案します。**

#### 即座に実施（Phase 1）: 選択肢1の実装
1行の修正で問題を解決。

#### 短期（Phase 2）: 削除済みタスクの可視化
```javascript
// UIで「削除済みタスクを表示」トグルを追加
function getFilteredTasks(includeDeleted = false) {
  return projectData.tasks.filter(task => {
    if (!includeDeleted && task.deleted) return false;
    // ... 既存のフィルタロジック
  });
}
```

#### 中期（Phase 3）: タスク復元機能
```javascript
function restoreTask(wbsNo) {
  const task = projectData.tasks.find(t => t.wbs_no === wbsNo && t.deleted);
  if (task) {
    task.deleted = false;
    task.deleted_at = null;
    saveToLocalStorage();
    renderTaskTable();
  }
}
```

**この段階的アプローチが最も優れている理由**:
- Phase 1だけで問題は解決（最小投資・最大効果）
- Phase 2で歯抜けの懸念も解消
- Phase 3で選択肢2の「番号再利用」ニーズにも対応（復元すれば欠番が埋まる）
- 各Phaseは独立して実装可能で、どこで止めてもシステムは健全

### Q6. 歯抜け番号

**回答: 小規模プロジェクトでは実務上問題になりません。**

理由：

1. **削除頻度が低い**: 1-5人のチームで3ヶ月〜1年の短期プロジェクトでは、削除されるタスクは全体の5-15%程度。歯抜けが目立つほどにはならない
2. **前例の豊富さ**: GitHub Issuesでは欠番は日常的で、誰も問題にしていない
3. **WBS番号の目的**: 一意な識別であり、連続する番号による美観ではない
4. **説明コスト**: 「削除されたタスクの番号は欠番として残ります」の一文で済む

万一問題になる場合は、Phase 2の「削除済みタスク表示」機能で完全に解消できます。

### Q7. 小規模チーム向けの最適解

**回答: 小規模チームだからこそ「シンプルで堅牢な方式」を選ぶべきです。**

エンタープライズとの違い：

| 判断基準 | エンタープライズ | 小規模チーム（本プロジェクト） |
|---------|--------------|------------------------|
| デバッグリソース | 豊富（専門チーム） | 限定的（開発者=運用者） |
| バグの影響 | SLAで管理 | 開発の手が直接止まる |
| 複雑さの許容度 | 高（ドキュメント・教育で対応） | **低（シンプルさが命）** |
| 実装者の継続性 | チーム交代あり | 同一人物が保守することが多い |

小規模チームでは、**バグが出にくい・理解しやすい・保守しやすい**ことが最優先です。選択肢1はこの3つすべてを満たします。選択肢2が提供する追加のトレーサビリティは、小規模チームでは「あれば便利だが必須ではない」レベルであり、そのために6〜10時間の実装コストとデータ整合性リスクを負う合理性はありません。

### エッジケース（追加で確認すべき）

| # | エッジケース | 期待動作 | 注意点 |
|---|-----------|---------|--------|
| 1 | タスク0件（初回利用） | `{Phase}.1.1` が生成される | maxNumber = 0 → newNumber = 1 |
| 2 | 手動でWBS番号を編集した場合 | 手動番号も含めてmaxを計算 | `TEST.100.1`があれば`TEST.101.1` |
| 3 | WBS番号に非数値が含まれる場合 | `parseFloat`でNaNになり、maxNumberに影響しない | 既存ロジックで対処済み |
| 4 | 同一フェーズに`.1`と`.2`の両方がある場合 | 最終セグメントの最大値を使用 | 現在のロジックは最終セグメント（Minor）のみ見ているが、本来はMajor番号を見るべきでは？（下記補足参照） |

### 💡 補足：番号生成ロジックの潜在的問題

現在のコードを精査すると、`maxNumber`の算出対象が **最後のセグメント（Minor番号）** になっている点が気になります：

```javascript
const lastNum = parseFloat(parts[parts.length - 1]);
```

WBS番号が `TEST.{Major}.{Minor}` 構造の場合：
- `TEST.15.1` → parts = ["TEST", "15", "1"] → lastNum = **1**
- `TEST.3.5` → parts = ["TEST", "3", "5"] → lastNum = **5**

この場合、`maxNumber`はMinor番号の最大値になります。しかし生成時は：
```javascript
const generatedWbsNo = `${phase}.${newNumber}.1`;
```
と **Major番号** として使用しています。

例えば `TEST.2.5` のみが存在する場合、maxNumber=5 → `TEST.6.1` が生成されます。意図と異なる可能性があります。

**推奨**: 今回の修正とは別途、Major番号の抽出ロジックの正確性を検証することを推奨します：

```javascript
// Major番号を正しく取得する場合
const majorNum = parseInt(parts[1], 10); // "TEST.15.1" → 15
```

この問題は選択肢1・2のどちらを選んでも存在する根本的な問題であり、今回のスコープとは独立して対処すべきです。

## 7. 見落としている可能性のある観点

### 7.1 LocalStorageの容量制限
論理削除を続けるとLocalStorageが肥大化します（通常5〜10MBの制限）。長期運用時に「古い削除済みタスクのパージ」機能が必要になる可能性があります。選択肢1ではIDが不変なので、パージしてもIDの衝突は起きません（maxNumberは既に生成済みの番号から決まるため）。ただし、パージ後にデータが失われた番号とのギャップが永続化する点は留意が必要です。

### 7.2 複数ブラウザ/デバイス間の同期
file://プロトコルでLocalStorageを使う設計では、複数環境間でのデータ同期は想定外ですが、JSONファイルを介した手動同期時に、両環境で同じ番号が生成されるリスクがあります。これは選択肢1・2共通の課題ですが、選択肢1の方がJSONマージ時のコンフリクト解決がシンプルです。

### 7.3 `original_wbs_no` フィールドの検討
将来的にタスク復元機能を実装する場合、選択肢2では「元のWBS番号」を別フィールドに保存する必要があります。選択肢1ではWBS番号が不変なので、この追加フィールドは不要です。

---

## 8. 最終的な推奨

**選択肢1（削除済みID除外 + 全履歴考慮）を強く推奨します。**

推奨理由を一言で表すと：

> **「より少ないコードで、より高い品質と安全性を実現できる」**

これはソフトウェア設計の理想形です。選択肢2には「削除が目に見えて分かりやすい」という感覚的な魅力がありますが、それは `deleted: true` フラグと将来の「削除済みタスク表示」機能で十分にカバーできます。IDの不変性を犠牲にしてまで得るべきメリットではありません。
