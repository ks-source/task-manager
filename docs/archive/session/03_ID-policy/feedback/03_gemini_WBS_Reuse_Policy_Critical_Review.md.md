
# WBS番号再利用ポリシー 再評価と補完提言

**作成日**: 2026-03-22  
**対象**: Task Manager - WBS Visualizer v2.13.4  
**目的**: 既存評価の妥当性を批判的かつ客観的に再点検し、必要に応じて新たな視点・補完提言を加える

---

## 0. エグゼクティブサマリー

先の総合評価は、**短期的な実装判断としては概ね妥当**です。  
特に、現状のデータ構造・実装制約・小規模チーム向けという前提では、**選択肢1（削除済みID除外 + 全履歴考慮）を採用する判断は現実的かつ安全寄り**です。

一方で、批判的に見ると、先の評価には次のような**補正・補完すべき点**があります。

### 妥当だと考える点
- **選択肢2（`_del`付与）は中長期的に不利**という判断
- **参照整合性の観点からID変更は危険**という指摘
- **小規模チームでは複雑さを増やさない方がよい**という判断
- **当面の対処としては選択肢1が最も費用対効果が高い**という結論

### 補正したい点
- 先の評価はやや **「WBS番号 = 不変な識別子」前提に寄りすぎている**
- 実務上は、**不変であるべきなのは内部ID**であり、**WBS番号は表示・構造用として再編されうる**ケースもある
- したがって、本質的な論点は  
  **「削除済みWBS番号を再利用するか」ではなく、  
  「WBS番号を参照キーとして使い続ける設計を維持すべきか」**  
  にあります

### 新たな提言
**短期は選択肢1、ただし中期的には「内部不変ID」と「表示用WBS番号」を分離する第3案を推奨**します。

#### 総評
- **実装判断として**: 妥当
- **設計原則として**: 一部再整理が必要
- **将来拡張の観点**: 第3案を明示すべき

## 2. 批判的に見たときの補正ポイント

### 2.1 「WBS番号の不変性」をやや強く言いすぎている

先の評価では、WBS番号をGitHub Issue番号やJiraのIssue IDに近いものとして扱っています。  
ただし、ここは**少し注意が必要**です。

#### 整理
- GitHub Issue番号やJira Issue Keyは、基本的に**識別子**
- WBS番号は、識別子であると同時に**構造・順序・計画の表現**
- したがってWBS番号は、純粋な内部IDとは性質が異なる

#### 実務上の含意
プロジェクト管理の世界では、WBSは計画の見直しに伴って**再採番**されることがあります。  
つまり、

- **内部IDは不変であるべき**
- **WBS番号は変更しうるが、変更管理されるべき**

という整理の方が、より一般化された設計原則です。

#### ただし本件では
現行システムでは `predecessors` などがWBS番号を参照しているため、  
**現時点ではWBS番号が事実上の主キーになっている**とみなさざるを得ません。

そのため、現行設計のままなら、

> 「WBS番号を変えない方が安全」

という先の結論はやはり妥当です。

### 2.2 本当の問題は「削除済み番号の再利用」ではなく「WBS番号を参照キーにしていること」

これは最も重要な補足視点です。

現状の議論は、

- 選択肢1: 再利用しない
- 選択肢2: `_del` に変えて再利用する

という比較ですが、これはどちらも  
**WBS番号がシステム内参照の中心である**ことを前提にしています。

しかし、より根本的には次のように考えるべきです。

#### 本質的な設計論点
- `wbs_no` を **表示用・構造用の番号**に留める
- 別に不変の `task_id` を持つ
- `predecessors` や `flowchart.nodeIds` は `task_id` を参照する

この分離ができれば、

- WBS番号の再採番
- 削除済みタスクの可視化
- タスク復元
- JSONインポート時の整合チェック
- 将来の複数プロジェクト統合

がすべてやりやすくなります。

#### したがって
先のレポートは**短期判断として正しい**一方で、  
**中長期のあるべき姿**としてこの第3案をもっと強く提言すべきでした。

### 2.3 選択肢1のリスクは「かなり低い」が、「2/10」は少し楽観的

先のレポートでは選択肢1のリスクを **2/10** としていました。  
方向性として低リスクなのは賛成ですが、数値としてはやや楽観的です。

#### 理由
選択肢1自体は単純でも、実際には次の不確実性があります。

1. **現在の採番ロジック自体に別バグの疑いがある**  
   - Major/Minor番号の取り扱いが曖昧
2. **手動編集されたWBSとの整合ルールが未確定**
3. **将来の復元・削除済み一覧・完全削除との整合ポリシーが未定**
4. **JSONインポートで過去データに揺れがある場合の扱いが未定**

#### 補正案
- 選択肢1: **3〜4/10**
- 選択肢2: **7〜8/10**

くらいの方が、実務上の感覚には近いです。

### 2.4 「実装20分」はコード変更時間であって、意思決定・QA込みではない

先の評価の「15〜20分」は、**コード差分だけ見ると妥当**です。  
ただし、意思決定資料としては、次のように書く方が正確です。

#### より現実的な見積もり
- **実装**: 10〜20分
- **手動確認**: 20〜40分
- **テストケース整理**: 20〜40分
- **変更履歴・説明追記**: 10〜20分

**合計**: 約1〜2時間

小規模ツールではこの程度でも十分軽量ですが、  
「20分で終わる」と断言すると、期待値だけ上げてしまう可能性があります。

### 2.5 「歯抜け番号は問題ない」は概ね正しいが、ユースケース条件付き

この点も大筋では賛成です。  
ただし、以下の場合は問題化しやすくなります。

#### 歯抜けが気になりやすいケース
- WBSが**成果物一覧**としても使われている
- 顧客説明やレビュー時に**連番の整然さ**を重視する
- 紙やPDFに出力して、**チェックリスト的に使う**
- 小規模でも、作業者が非エンジニア中心

#### 逆に問題になりにくいケース
- 開発者中心の運用
- WBS番号を主に検索・参照用として使う
- 削除よりも「無効」「中止」「保留」が多い運用
- 削除済み一覧や復元機能を将来持つ前提

したがって、  
**「歯抜けは一般には許容可能。ただし、帳票・説明資料用途が強いなら補助UIが必要」**  
と表現するのがより客観的です。

## 3. 新たに加えるべき視点

### 3.1 第3案: 内部IDとWBS番号の分離

これが最も重要な追加視点です。

#### 提案
各タスクに以下を持たせます。

- `task_id`: 内部用の不変ID
- `wbs_no`: 表示用のWBS番号
- `deleted`: 論理削除フラグ
- `deleted_at`: 削除日時

#### 参照の持ち方
- `predecessors`: `task_id` の配列
- `flowchart.nodeIds`: 可能なら `task_id` ベースへ移行
- UI表示時のみ `wbs_no` を見せる

#### これにより可能になること
- WBS番号の変更や再採番をしても参照が壊れない
- 削除済み番号の扱いを柔軟に変えられる
- 復元時にも安全
- JSONインポート時のマージ判定がしやすい
- 将来的な複数プロジェクト統合に強い

#### デメリット
- 今すぐは少し重い
- 既存データの移行が必要
- UI/検索/入出力の修正範囲が広がる

#### 結論
**今すぐ必須ではないが、将来拡張を見据えるなら最優先で意識すべき設計改善**です。

### 3.2 「削除」より「アーカイブ / キャンセル」の方が業務的に自然ではないか

プロジェクト管理では、削除よりも次の状態がよく使われます。

- Active
- Completed
- Cancelled
- Archived
- Deleted（管理用）

#### 理由
タスクは実務上、消えるよりも「やらないことになった」「無効になった」ケースが多いからです。

#### 本件への示唆
もしユーザーが本当に求めているのが  
「消したい」のではなく「今後は見たくない」  
であれば、論理削除よりも、

- `status: cancelled`
- UI上は非表示
- 必要なら表示切替

の方が自然です。

#### 意味
この場合、
- WBS番号はそのままでよい
- 歯抜けも説明しやすい
- 監査証跡も保持しやすい
- 復元も容易

よって本件は、  
**削除ポリシーの議論であると同時に、状態管理設計の議論でもある**  
と捉えるべきです。

### 3.3 「採番ポリシー」を仕様として固定すべき

今回の問題は、単にコード修正の問題ではなく、  
**番号生成仕様が曖昧なこと**が原因の一部です。

明文化すべき項目は少なくとも以下です。

#### 決めるべき仕様
1. WBS番号は**手動変更可能か**
2. 自動生成時は **phaseごとのMajor採番** か
3. `Minor` は常に `1` から始まるのか
4. 削除済みタスクは採番計算に含めるか
5. 復元時に元番号へ戻すか
6. 完全削除時に再利用を認めるか
7. JSONインポート時に重複があればどうするか

#### コメント
ここが曖昧だと、選択肢1を入れても将来また同種の議論が再発します。

### 3.4 file:// + LocalStorage + JSON運用では「マージ競合」が本質リスク

これは先のレポートで少し触れられていましたが、重要なので補強します。

#### 想定シナリオ
- PC A で `TEST.15.1` を作成
- PC B でも別途 `TEST.15.1` を作成
- 後でJSONを取り込む

このとき、削除ポリシー以前に  
**オフライン分散更新でWBS競合**が起きます。

#### 意味
将来的な複数環境利用を考えるなら、  
WBS採番問題は「削除番号再利用」よりも  
**同時編集・分岐データ統合**の方が大きな課題になりえます。

#### 補完提言
将来の複数環境利用を少しでも想定するなら、
- 内部ID導入
- import時の重複検出
- conflict report出力
を視野に入れるべきです。

## 5. 最終提言

### 5.1 短期判断

**直近の修正としては、選択肢1を採用するのが妥当**です。

理由:
- 現行構造を崩さない
- リスクが低い
- 参照破壊を起こさない
- 小規模運用に合う

### 5.2 中期判断

**ただし、設計としては「WBS番号を主キー的に使っている」状態から脱却する方が望ましい**です。

#### 推奨する段階的ロードマップ

##### Phase 1: 今すぐ
- 選択肢1を導入
- 採番ロジックのMajor/Minor解釈を確認
- テスト追加

##### Phase 2: 近いうち
- 削除済み表示トグル
- 復元機能
- 「削除」ではなく「キャンセル/アーカイブ」導線の検討

##### Phase 3: 設計改善
- `task_id` 導入
- 参照を `task_id` ベースへ移行
- `wbs_no` を表示用へ格下げ

##### Phase 4: 余力があれば
- 再採番機能
- import競合検知
- 監査ログ強化

### 5.3 選択肢2の位置づけ

`_del` サフィックス案は、  
**「JSONを人間が直接読むと削除済みだと分かりやすい」**  
という一点では分かりやすさがあります。

ただし、それは
- `deleted: true`
- `deleted_at`
- 削除済み一覧UI

で代替可能です。

そのため、設計・整合性・将来保守まで含めると、  
**採用理由としては弱い**です。

## 7. 最終結論

先の総合評価は、**実装意思決定としては十分に妥当**です。  
無理にひっくり返す必要はありません。

ただし、より批判的・客観的に整理すると、最も重要な追加視点は次の1点です。

> **本件の本質は「削除済みWBS番号を再利用するか」ではなく、  
> 「WBS番号を識別子と表示番号のどちらとして扱うのか」を整理できていないことにある。**

この整理に立つと、提言は次のようにまとめられます。

#### 最終提言
1. **短期は選択肢1を採用**
2. **選択肢2は採用しない**
3. **中期的には内部不変IDの導入を検討**
4. **削除よりも状態管理（cancelled / archived）設計を見直す**
5. **採番ロジックと採番仕様を明文化する**
