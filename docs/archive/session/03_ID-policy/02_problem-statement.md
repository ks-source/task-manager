# 問題の定義

**発見日**: 2026-03-22
**報告者**: ユーザー
**重要度**: 中
**影響範囲**: WBS番号の一意性、データ整合性

---

## 📋 問題の概要

論理削除機能の実装後、新規タスク作成時にWBS番号を自動生成すると、**削除済みタスクのWBS番号が再び提案される**問題が発見されました。

---

## 🗣️ ユーザーからの報告（原文）

> 削除機能は改善されましたが、その過程で下記の問題を発見しました。
> 新規タスク作成画面で新規タスクに割り当てる「WBS番号」自動生成時に、上記手順で削除したタスクのIDが割り当てられ、手動で新しい番号を割当てる必要が発生しているようです。

---

## 🔍 具体的な再現シナリオ

### **シナリオ1: 全削除後の番号リセット**

**手順**:
1. TEST.1.1 を作成
2. TEST.2.1 を作成
3. TEST.3.1 を作成
4. TEST.1.1, TEST.2.1, TEST.3.1 をすべて削除（論理削除）
5. 新規タスク作成で「WBS番号自動生成」ボタンをクリック

**現状の動作**:
- 生成される番号: `TEST.1.1`

**期待される動作**:
- 生成される番号: `TEST.4.1`（削除済みを含む全履歴から連番）

**問題点**:
- 削除済みタスクと同じ番号が再利用される
- ユーザーが手動で `TEST.4.1` に修正する必要がある

---

### **シナリオ2: 特定番号削除後の再利用**

**手順**:
1. TEST.15.1 を作成
2. TEST.15.1 を削除（論理削除）
3. 新規タスク作成で「WBS番号自動生成」ボタンをクリック

**現状の動作**:
- phaseTasksが空配列 → maxNumber = 0
- 生成される番号: `TEST.1.1`

**期待される動作**:
- 生成される番号: `TEST.16.1` または `TEST.15.2`

**問題点**:
- 削除済みタスクの番号履歴が無視される
- 番号が1から再開される

---

### **シナリオ3: 中間番号削除による歯抜け**

**手順**:
1. TEST.1.1, TEST.2.1, TEST.3.1, TEST.4.1, TEST.5.1 を作成
2. TEST.2.1, TEST.4.1 を削除（論理削除）
3. 現在のタスク一覧: TEST.1.1, TEST.3.1, TEST.5.1（歯抜け）
4. 新規タスク作成で「WBS番号自動生成」ボタンをクリック

**現状の動作**:
- phaseTasks = [TEST.1.1, TEST.3.1, TEST.5.1]
- maxNumber = 5
- 生成される番号: `TEST.6.1` ✅

**期待される動作**:
- 同上（このケースは正常動作）

**補足**:
- 番号の歯抜け（TEST.2, TEST.4 がない）は発生するが、再利用はされない
- ユーザーから「なぜTEST.2がないのか？」という疑問が出る可能性

---

## 📊 問題発生のメカニズム

### **原因のコードフロー**

```javascript
// task-manager.html:3330
const phaseTasks = projectData.tasks.filter(t => t.phase === phase && !t.deleted);
//                                                                     ^^^^^^^^
//                                                  削除済みタスクを除外

// task-manager.html:3333-3343
let maxNumber = 0;
phaseTasks.forEach(t => {  // ← 削除済みが含まれない配列
  const parts = t.wbs_no.split('.');
  if (parts.length > 1) {
    const lastNum = parseFloat(parts[parts.length - 1]);
    if (!isNaN(lastNum) && lastNum > maxNumber) {
      maxNumber = lastNum;  // ← 未削除タスクのみから最大値を算出
    }
  }
});

// task-manager.html:3346
const newNumber = Math.floor(maxNumber) + 1;  // ← 削除済みタスクを考慮していない
```

**問題の核心**:
- `phaseTasks` から削除済みタスクを除外
- その除外された配列から `maxNumber` を計算
- 結果、削除済みタスクの番号が「存在しなかった」かのように扱われる

## 🎯 影響範囲

### **1. データ整合性への影響**

**WBS番号の一意性**:
- 同じWBS番号を持つタスクが複数存在しうる
- 例: TEST.15.1（削除済み）と TEST.15.1（新規）

**JSONエクスポートへの影響**:
```json
{
  "tasks": [
    {"wbs_no": "TEST.15.1", "task_name": "旧タスク", "deleted": true},
    {"wbs_no": "TEST.15.1", "task_name": "新タスク", "deleted": false}
  ]
}
```
- WBS番号の重複が発生
- JSONインポート時に識別が困難

---

### **2. 参照整合性への影響**

**predecessorsフィールド**:
```json
{
  "wbs_no": "TEST.16.1",
  "predecessors": "TEST.15.1",  // ← どちらのTEST.15.1を参照？
  "deleted": false
}
```
- 削除済みタスクを参照しているのか、新規タスクを参照しているのか不明

---

### **3. ユーザー体験への影響**

**手動修正の負担**:
- 自動生成された番号が不適切
- 毎回手動で修正が必要
- 「自動生成」の価値が低下

**混乱の可能性**:
- 「なぜTEST.2がないのか？」
- 「削除したタスクと同じ番号が出てくる」
- トレーサビリティの低下

---

## 💡 ユーザーが提案した2つのアプローチ

### **提案1: 削除済みID除外 + 全履歴考慮**
> 論理削除したタスクのWBS番号は意図的に除外して自動生成する

**解釈**:
- 削除済みタスクの番号を再利用しない
- 全タスク（削除含む）の履歴から次の番号を生成
- 欠番は残るが、番号の再利用は避ける

---

### **提案2: `_del` サフィックス付与**
> 論理削除したタスクのWBS番号へ論理削除を示す「del」などを付与（例：TEST.15.1_del 等）して明示的に論理削除済みであることを示す

**解釈**:
- 削除時にWBS番号を変更
- `TEST.15.1` → `TEST.15.1_del`
- 削除済みタスクが明示的に可視化される
- 新規タスクは `TEST.15.1` を使用可能（再利用）

## 📝 問題の本質

### **WBS番号の性質**

1. **ユーザー可視ID**:
   - システム内部IDではなく、ユーザーが直接見る・参照する
   - ドキュメント、レポート、会議で使用される

2. **階層構造**:
   - `{Phase}.{Major}.{Minor}` の構造
   - 順序性を持つ（連番）

3. **不変性の期待**:
   - 一度割り当てられた番号は変わらない（ユーザー期待）
   - 削除後も「その番号は使われた」という履歴が残る

---

### **トレードオフ**

| 観点 | 選択肢1（除外のみ） | 選択肢2（サフィックス） |
|-----|------------------|---------------------|
| **実装コスト** | 低 | 中～高 |
| **トレーサビリティ** | 中（欠番として残る） | 高（明示的マーク） |
| **データ整合性リスク** | 低 | 中～高（参照破壊） |
| **ユーザー混乱** | 中（歯抜け） | 低（明確） |
| **業界標準準拠** | 高 | 低 |
