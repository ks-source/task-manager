## 🔍 検討している2つのアプローチ

### **選択肢1: 削除済みID除外 + 全履歴考慮**
削除済みタスクのWBS番号を再利用せず、全タスク（削除含む）の履歴から次の番号を生成する。

👉 詳細: [03_option-1-exclude-only.md](./03_option-1-exclude-only.md)

### **選択肢2: `_del` サフィックス付与**
削除時にWBS番号を `{元番号}_del` に変更し、削除されたことを明示的にマークする。

👉 詳細: [04_option-2-suffix-approach.md](./04_option-2-suffix-approach.md)

## 📊 プロジェクトのコンテキスト

### **システム概要**
- **名称**: Task Manager - WBS Visualizer
- **種別**: ブラウザベースの単一HTMLファイルアプリケーション
- **プロトコル**: file:// (サーバー不要)
- **データ永続化**: LocalStorage + File System Access API
- **ユーザー層**: 個人開発者～小規模チーム（1-5人）
- **運用期間**: 短期プロジェクト（3ヶ月～1年）が主

### **WBS番号の構造**
```
{Phase}.{MajorNumber}.{MinorNumber}

例:
- TEST.15.1
- RELEASE.7.2
- DESIGN.3.5
```

### **論理削除の実装**
- タスクに `deleted: true` フラグを設定
- `deleted_at` にタイムスタンプを記録
- データは保持（JSONエクスポートに含まれる）
- UIでは非表示（フィルタリング）

### **現在の課題**
削除後に同じWBS番号が新規タスク作成時に提案される可能性がある（現在の実装では削除済みタスクを考慮していない）

## ❓ 具体的な質問

### **Q1. ID再利用の是非**
プロジェクト管理ツールにおいて、削除されたタスクのIDを再利用することの是非について、業界標準の観点からどう評価しますか？

### **Q2. WBS番号の特殊性**
WBS番号は階層構造を持つユーザー可視IDですが、通常のシーケンシャルIDとは異なる考慮が必要でしょうか？

### **Q3. 参照整合性**
選択肢2（サフィックス付与）では、predecessorsフィールドなどの参照が破壊される懸念がありますが、この懸念は妥当でしょうか？また、対処方法はありますか？

### **Q4. トレーサビリティ**
監査証跡やトレーサビリティの観点から、どちらのアプローチがより適切でしょうか？

### **Q5. 代替案**
提示した2つの選択肢以外に、より優れたアプローチはありますか？

### **Q6. 歯抜け番号**
選択肢1では番号に歯抜けが発生しますが（例: 1, 3, 5のみ残る）、これはプロジェクト管理上問題になりますか？

### **Q7. 小規模チーム向けの最適解**
1-5人の小規模チームでの利用を想定した場合、エンタープライズ向けとは異なる判断基準がありますか？
